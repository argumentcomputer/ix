Status: *design / RFC*. Pre-implementation. This document states a problem with
IxVM's memoization and proposes a fix: annotate each de Bruijn variable with its type so
inference, WHNF, and definitional equality become *context-free* (=Expr -> Expr=). It
records the adversarial soundness analysis (a substitution-preservation proof, an
index-only-def-eq argument, and a full audit of where the one-shot validator must run).

*IMPLEMENTED* on branch =gb/typed-bvar= (Stages A–E below). The full =ixvm= kernel suite
passes context-free (=lake test -- --ignored ixvm=, 303/303): =Nat.add_comm=, every
primitive, decidability, and all recursors (incl. mutual =Even.rec=/=Odd.rec=)
type-check. =types= is now empty on every path; the =ctx_trim= subsystem is deleted; the
memo key is the expression pointer(s) alone. *Measured cost*: a modest FFT increase from
annotation size (e.g. =Nat.add_comm= +3.5%, =Nat.decLe= +10%, =nat_div_lit= +11%; a few
terms slightly negative) — confirming open question #1: the win is *architectural*, not a
raw cost reduction (see [[#open-questions]]).

An earlier draft of this document proposed a different mechanism ("FVar opening" — open
binders into fresh free variables). That idea is *scrapped*; it survives only as a
[[#alt-fvar][rejected alternative]]. The design below — *typed BVars* — is simpler, has
no special-cased comparison logic, and matches the existing kernel architecture.
#+end_quote

* TL;DR

IxVM typechecks Lean-4 core terms inside the first-order Aiur zkDSL. Aiur turns every
function call into a logUp lookup keyed on the *tuple of its argument pointers*, with
multiplicities — so identical calls are memoized "for free." That memoization is the
whole performance story of the VM.

Today the three core kernel functions all thread a *local typing context*:

#+begin_src
k_infer    (e,    types, top, addrs) -> KExpr
whnf       (e,    types, top, addrs) -> KExpr
k_is_def_eq(a, b, types, top, addrs) -> G
#+end_src

=types : List‹KExpr›= is the list of binder types (innermost-first); =top= / =addrs= are
*pinned* for an entire run. So the effective memo key is =(e_ptr, types_ptr)=, and because
=types= changes with binder depth/path, *the same subterm re-infers under every context it
appears in*. (There is already a partial antidote, =ctx_trim=, which makes *closed*
subterms share — see [[#types-poison]].)

*The proposal.* Change the variable node from =BVar(idx)= to =BVar(idx, type)=, where
=type= is the variable's type *valid at the occurrence* (= the binder's declared type
lifted by =idx+1=). Then:

- Ingress computes each annotation while it still has the binder-type context in hand.
- A single =wellFormed= pass per constant body — an *extension of the kernel's existing
  =validate_expr_well_scoped= pass* (=Check.lean:119=) — validates the annotations (and
  that there are no loose BVars) so inference may *trust* them.
- =infer(BVar(i, ty)) = ty=. No context lookup. And then =infer= / =whnf= / =k_is_def_eq=
  / the inductive checks *all drop =types=* and become functions of self-contained,
  self-annotated expressions — keyed on the expression pointer alone.

*Why this is the right shape (verified):*
- *Sound*, with two small conditions ([[#correctness]]). The load-bearing lemma —
  substitution transports annotations correctly — is proven, reduced to the standard
  lift/lift and inst/lift commutation laws, with a worked dependent example. The
  annotation is just a *memoized copy of the type the current kernel already computes*
  (=types_lookup= already returns =lift(binder_type, i+1, 0)=), so this is
  memoize-a-pure-function, the strongest soundness footing.
- *No special comparison logic.* =def_eq='s =BVar=–=BVar= arm stays plain index equality
  and *ignores* the annotation — provably sound and in fact *more complete* than comparing
  it (immune to post-reduction annotation drift). None of the FVar idea's joint-opening
  conditions are needed.
- *No memoization regression.* de Bruijn indices are outer-invariant, so a reused open
  subterm keeps sharing across depths (the FVar scheme lost this). Sharing *matches*
  =ctx_trim= exactly, while the context argument and the whole =ctx_trim= subsystem are
  deleted.
- *The wire format is unchanged.* Annotations are recomputed at ingress, never serialized.
- The cost is real but localized: annotations make terms *bigger*, and substitution must
  recurse into them. Measure it ([[#open-questions]]).

* Background: how Aiur memoizes, and why =types= hurts
:PROPERTIES:
:CUSTOM_ID: background
:END:

** The memo mechanism is argument-keyed logUp

In Aiur a =Call= is a logUp into a per-callee =QueryMap= (=src/aiur/querymap.rs=). The
key is the flat tuple of the call's *argument field elements*; the value is
=(output, multiplicity)=. At execution (=src/aiur/execute.rs:280-301=) a call probes
=function_queries[callee].get(&args)=: on a *hit* it bumps the multiplicity and reuses the
stored output — *the callee does not re-execute*; on a *miss* it records a new row.

Because =KExpr=, =List‹KExpr›=, =Addr= are content-addressed via =store= (Blake3 deduped
to a canonical pointer), structurally-equal values *are* the same field element. So "two
calls share" ⟺ "their argument-pointer tuples are identical."

** Aiur's cost model: arguments are cache surface, accumulators work against you
:PROPERTIES:
:CUSTOM_ID: cost-model
:END:

A principle that motivates this entire refactor — and that already shapes how the IxVM
kernel is written — runs *opposite* to ordinary intuition about efficient code:

#+begin_quote
On a normal machine, a function with more arguments — threading transient state as
*accumulators* — is usually /faster/: it avoids recomputation and lets the compiler reuse
a single stack frame (tail calls). On Aiur the trade-offs invert. *Each argument widens
the memo key*, so it enlarges the lookup "space" and makes collisions (cache hits) rarer:
fewer arguments ⇒ more calls coincide ⇒ more sharing. And *tail recursion buys nothing* —
every call frame is recorded in the execution trace; there is no frame popping or
overwriting, all frames must coexist. So "stack depth is free" but "argument width is
not": an accumulator that would /save/ work on a CPU instead *bloats the key and blocks
sharing* on Aiur.
#+end_quote

This is not hypothetical; the kernel already encodes it. =collect_spine=
(=KernelTypes.lean:56-69=) is written *without* an accumulator on purpose:

#+begin_quote
"Non-tail (no accumulator): keyed on =e= alone, so Aiur memoization dedups shared
sub-spines across reductions. An accumulator would thread =acc= through the memo key and
block that sharing (tail recursion buys nothing in Aiur — stack depth is free).
=list_snoc= keeps order and is itself memoized."
#+end_quote

*Consequence for normalization strategy — why substitution, not NbE.* Through this lens,
normalization-by-evaluation /is/ the "accumulate transient work in an argument"
transformation: the *environment* is a lazy, suspended substitution carried beside the
term, and actual normalization only happens at *quote* / readback. That environment is an
accumulator — on Aiur it lands *in the memo key*, re-poisoning sharing at the value level
(an NbE closure's identity is =(body, env)=, exactly the =(expr, context)= pair we are
trying to escape). So NbE's usual advantage largely *evaporates* under Aiur's cost model,
and can even invert: plain *substitution* recomputes more (worse on a CPU) but threads
*no* accumulator, so each =whnf= / =infer= call keys on the bare expression and shares
maximally (better on Aiur). NbE is thus "not really better, if at all" here — which is why
IxVM normalizes by substitution and why the dead =KValNode= NbE substrate ([[#prior-art]])
was never the right road.

The =types= context is the *same* anti-pattern inside =infer= / =whnf= / =def-eq=: a
transient side argument threaded through the recursion. *This whole refactor is one
application of the cost-model principle — drop the accumulator argument, make the term
self-describing so the work can be recomputed-and-memoized on the expression alone.* A
typed =BVar= is the device that lets us remove the argument without losing what it
carried: the variable's type rides /inside/ the term, not /beside/ it.

** =types= is the poison; =ctx_trim= is the current antidote
:PROPERTIES:
:CUSTOM_ID: types-poison
:END:

Since =top= / =addrs= are pinned, the live key for =k_infer= / =whnf= is
=(e_ptr, types_ptr)=. =types= varies with binder depth and path, so the same =e=
re-infers per context. Today =infer(BVar(i)) = types_lookup(types, i)=
(=Infer.lean:66=), and =types_lookup= returns =expr_lift(types[i], i+1, 0)=
(=Infer.lean:34-45=) — *the very "occurrence-valid" type this refactor proposes to cache
in the node.*

There is already a mitigation. Each entry point applies ~ctx_trim(types, expr_lbr(e))~
(=Subst.lean:119-128=, called at =Infer.lean:58-61=, =Whnf.lean:320=, =DefEq.lean:48=),
keeping only the suffix of =types= reachable from =e='s loose-bvar range:

- A *closed* subterm (=expr_lbr(e) == 0=) trims to =Nil= and already keys to =(e, Nil)= —
  fully shared across depths.
- An *open* subterm keys to =(e, trimmed_suffix_ptr)= — still a context-dependent pointer.

So =ctx_trim= solves the closed case. The proposal makes *open* subterms self-contained
too, deletes =ctx_trim=, and — the structural prize — lets =whnf= / =def_eq= / the
inductive checks shed =types= entirely.

* The proposal: typed BVars + a =wellFormed= validator
:PROPERTIES:
:CUSTOM_ID: proposal
:END:

** The node

Change one constructor in =KExprNode= (=KernelTypes.lean:40=):

#+begin_src
BVar(idx: G)            ⟶            BVar(idx: G, type: KExpr)
#+end_src

** The annotation is the /occurrence-valid/ type

For a =BVar(i, ty)= occurring under scope context Γ = [τ₀, …, τ_{d-1}] (innermost-first):

#+begin_src
ty  ==  lift(Γ[i], i+1, 0)
#+end_src

That is: the binder type =Γ[i]= (as written at the binder, where it is valid under a
context =i+1= frames shorter) *lifted up by =i+1=* to be valid at the occurrence. This is
*exactly* what =types_lookup= computes today (=Infer.lean:39-42=, which does =i+1=
successive =lift(·,1,0)= calls — definitionally equal to one =lift(·, i+1, 0)=). So we are
caching, on the node, a value the kernel already produces and trusts.

*Store the lifted type, not the raw binder type.* This is forced, not stylistic. The raw
binder type is valid at the binder's *absolute depth* =d−i−1=; to transport it under
substitution you'd need to know =d= (the whole context depth) — the very thing we're
deleting. The lifted/occurrence-valid type lives at *exactly the BVar's own scope*, so it
is an ordinary same-scope subexpression and transforms under the *local* =lift= / =inst1=
by the same recursive call (see [[#subst-lemma]]). Annotations *nest*: the annotation is
itself an expression whose own =BVar=s carry annotations; this bottoms out because each
annotation is a lift of an existing binder type, finite in the term's binder depth.

** Ingress computes the annotation

Annotations are born at ingress, where the binder-type context is naturally in scope.
Today =Convert.lean:155-156= transliterates a variable with *no* context:

#+begin_src
Expr.Var(idx) => store(KExprNode.BVar(flatten_u64(idx)))
#+end_src

The refactor threads a =types: List‹KExpr›= through =convert_expr= (=Convert.lean:140-209=,
which currently does *not* carry one), pushing each binder type on descent into
=Lam=/=Forall=/=Let= bodies, and emits =BVar(idx, lift(types[idx], idx+1, 0))=. This is the
*single creation point* for annotations.

** =wellFormed=: extend the validator the kernel already runs

The kernel *already* runs a once-per-constant structural pass before inference:
=validate_expr_well_scoped(e, depth, bound, top)= (=Check.lean:119-148=) threads a =depth=
counter, asserts =BVar(i) < depth= (the *no-loose-BVars* check, =Check.lean:123=), checks
=Const= universe arity, and crosses binders with =depth+1=. It is invoked via
=validate_const_well_scoped= (=Check.lean:152=) from =check_const= (=Check.lean:204=),
*before* any inference. (Same shape in Rust: =src/ix/kernel/check.rs:423-515=.)

=wellFormed= is this pass, *extended to also thread the binder-type context and check the
annotation*:

#+begin_src
wf(BVar(i, ty), Γ):  assert i < |Γ|                          -- no loose bvars (already there)
                     assert ty ≡ptr lift(Γ[i], i+1, 0)       -- NEW: canonical annotation
                     wf(ty, Γ)                                -- annotation itself well-annotated
wf(Lam/Forall(τ,b), Γ): wf(τ, Γ); wf(b, τ::Γ)
wf(Let(τ,v,b), Γ):      wf(τ, Γ); wf(v, Γ); wf(b, τ::Γ)
wf(App(f,a), Γ):        wf(f, Γ); wf(a, Γ)
wf(Proj(_,_,e), Γ):     wf(e, Γ)
Srt/Const/Lit:          ok
#+end_src

Entry =wf(body, [])= enforces top-level closedness (a constant body has no free vars). The
check is *structural / pointer* equality against the recomputed canonical lift — *not*
=def_eq= (see [[#wf-structural]]). It calls no =whnf= and no =def_eq=: a single linear,
context-threading, memoized walk. *This is the one place the context still lives — and it
is allowed to, because it runs once per constant and is not on the memoized hot path.*

Its purpose is *trust*: the term (and its annotations) may be adversarial prover-supplied
witness data. Once =wellFormed= passes, inference may rely on =BVar= annotations being
canonical.

** Inference, rewritten

With trusted annotations, the binder arms of =k_infer_core= lose all context handling:

#+begin_src
BVar(i, ty)         => ty                                  -- O(1); no types_lookup, no lift
Lam(ty, body)       => let _ = k_ensure_sort(ty)
                       store(Forall(ty, k_infer(body)))     -- NO context push
Forall(ty, body)    => let u1 = k_ensure_sort(ty)
                       let u2 = k_ensure_sort(body)
                       store(Srt(imax(u1, u2)))
Let(ty, val, body)  => let _ = k_ensure_sort(ty)
                       let _ = k_check(val, ty)
                       k_infer(expr_inst1(body, val, 0))     -- zeta; unchanged
App, Const, Proj…   => as today, minus the `types` argument
#+end_src

=k_ensure_sort=, =k_check=, =whnf=, =k_is_def_eq= are all called *without* =types=. Note
=k_ensure_sort= still validates every binder type is a sort during inference — so binder
types are checked for real even though =wellFormed= only checked annotation *canonicality*
(the two checks compose: =wellFormed= makes annotations faithful to the binder types;
=k_ensure_sort= makes the binder types valid).

** Substitution must transport annotations
:PROPERTIES:
:CUSTOM_ID: subst-rules
:END:

Because the annotation is a same-scope subexpression, =expr_lift= / =expr_inst1= /
=expr_inst_many= recurse into it *at the same cutoff/depth as the =BVar= node* (never
=+1=; a =BVar= is a leaf, it introduces no binder):

#+begin_src
expr_lift_walk(BVar(i, ty), s, c):
  i <  c => BVar(i,     lift(ty, s, c))
  i >= c => BVar(i + s, lift(ty, s, c))

expr_inst1_walk(BVar(i, ty), arg, depth):
  i <  depth => BVar(i,     inst1(ty, arg, depth))
  i == depth => expr_lift(arg, depth, 0)          -- variable replaced; annotation discarded
  i >  depth => BVar(i - 1, inst1(ty, arg, depth))

expr_lbr(BVar(i, ty)) = max(i + 1, expr_lbr(ty))
#+end_src

=expr_inst_many= is the n-ary generalization. The =expr_lbr= change keeps the lift/subst
fast-paths sound (the fast path skips the walk when no loose bvar lies at/above the cutoff;
folding the annotation's loose range into =lbr= ensures a needed substitution into an
annotation is never skipped). =expr_inst_levels= (=Levels.lean=) must also recurse into the
annotation to instantiate universe params there.

* The cascade: =whnf=, =def_eq=, and the inductive checks go context-free
:PROPERTIES:
:CUSTOM_ID: cascade
:END:

=whnf= threads =types= *only* to feed nested =k_infer= / =is_prop_type= / =k_is_def_eq=
(the struct-eta Prop-guard =Whnf.lean:645-646= and the K-ctor synth/verify
=Whnf.lean:802-815=); its β/δ/ι/ζ/proj/primitive reduction is *purely syntactic* — there
is no =list_lookup(types, …)= anywhere in =Whnf.lean= or =Primitive.lean=. So once
=k_infer= / =k_is_def_eq= are context-free, =whnf= trivially becomes =whnf(e) -> KExpr=,
and =Primitive.lean='s dispatch chain follows. Same for the inductive checker.

=BVar= stays an inert neutral head in =whnf=: the fast path returns it unchanged
(=Whnf.lean:316=) and =whnf_with_spine='s catch-all already treats it as a stuck spine head
(=Whnf.lean:131=). =whnf= never reads the annotation (it is type-agnostic).

#+begin_quote
*Ordering:* the embedded calls at =Whnf.lean:645-646= and =:802-815= mean =whnf= cannot be
context-free before =k_infer= / =k_is_def_eq= are. Cut the mutually-recursive cluster over
together.
#+end_quote

* Correctness
:PROPERTIES:
:CUSTOM_ID: correctness
:END:

A constructive soundness analysis (independently re-deriving the lemmas from source) and a
separate red-team audit produced the verdict: *SOUND, with two explicit conditions.* The
argument has four parts.

** The well-annotatedness invariant (WA)

For =t= under scope Γ, =WA_Γ(t)= holds iff every =BVar(i, ty)= in =t= (read under the Γ
extended by the binders crossed to reach it) satisfies =i < |Γ|= and
=ty ≡ptr lift(Γ[i], i+1, 0)= and =WA_Γ(ty)=, with the obvious structural clauses for the
other nodes (pushing the binder type for =Lam=/=Forall=/=Let= bodies). =wellFormed= is
exactly the decision procedure for =WA_[](body)=.

** The substitution-preservation lemma (proven)
:PROPERTIES:
:CUSTOM_ID: subst-lemma
:END:

#+begin_quote
*Lemma.* With the [[#subst-rules][rules above]] (annotation transported at the node's own
cutoff/depth): if =WA_{Δ++Γ}(t)= then =WA_{Δ++Σ++Γ}(expr_lift(t, |Σ|, |Δ|))=; and if
=WA_{Δ++[τ]++Γ}(t)= and =WA_Γ(arg)= then =WA_{Δ'++Γ}(expr_inst1(t, arg, |Δ|))=.
#+end_quote

Proof reduces, by case on the =BVar= index vs the cutoff/depth, to the standard de Bruijn
*lift/lift* and *inst/lift* commutation laws:

- =lift=, =i < c= (inner-bound, index fixed): annotation relifts by =lift(ty, s, c)= —
  correct by =lift(lift(u, i+1, 0), s, c) = lift(…, i+1, 0)= for =c ≥ i+1=. *Cutoff is =c=,
  the node's own.*
- =lift=, =i ≥ c= (free, index =i+s=): annotation's vars are all =≥ i+1 > c=, so
  =lift(ty, s, c)= shifts them all by =s=, giving =lift(Γ[i−c], i+s+1, 0)=. ✓
- =inst1=, =i == depth=: =BVar= *replaced* by =lift(arg, depth, 0)=; annotation correctly
  *discarded* (no surviving variable to annotate).
- =inst1=, =i < depth=: =BVar(i, inst1(ty, arg, depth))= — inst/lift commutation gives the
  canonical annotation for the eliminated-binder context.
- =inst1=, =i > depth=: =BVar(i−1, inst1(ty, arg, depth))= — all of =ty='s vars are
  => depth=, so they decrement by one in lockstep with the index. ✓

*Worked dependent example.* β-reducing =(λ(A:U). λ(x:A). λ(y:A). x) Nat=, fully annotated,
the body =x= is =BVar(1, BVar(2, U))= ("x, whose type A is index 2 from the body"). The
substitution =inst1(·, Nat, 0)= yields:

#+begin_src
λ(x:Nat). λ(y:Nat). BVar(1, Nat)
#+end_src

Both =A=-domain occurrences hit =i == depth= and are *replaced* by =Nat= (annotation
discarded); the body's =BVar(1, BVar(2,U))= hits =i < depth=, keeps index 1, and its
annotation =inst1(BVar(2,U), Nat, 2)= hits =i == depth= internally → =Nat=. Inferring the
result gives =Forall(Nat, Forall(Nat, Nat))= = =Nat → Nat → Nat=. Every annotation
transformed correctly; the discard/recurse split fired exactly where variables were
eliminated vs survived.

** =def_eq= ignores the annotation — sound /and/ complete

=def_eq='s =BVar=–=BVar= arm (=DefEq.lean:383-392=) compares *index only*, ignoring the
annotation. Both sides are compared at the *same* ambient scope (the binder arms recurse on
both bodies under the same pushed binder). Therefore:

- *Sound (no false positive).* In a fixed Γ, =BVar(i)= and =BVar(j)= denote the same
  variable iff =i == j=; the same variable has one type, so for well-annotated terms equal
  index ⟹ *pointer-equal* annotations. Accepting on =i == j= can never equate distinct
  variables. The annotation is /redundant/, not /ignored-at-peril/.
- *Complete (no false negative), strictly more than comparing it.* Reduction can produce
  two occurrence-valid-but-not-pointer-equal annotations for the same index on the two
  sides (different reduction paths). *Comparing* annotations structurally would spuriously
  reject =BVar(i) ≟ BVar(i)=; comparing them via =def_eq= would be sound but expensive and
  re-entrant. *Ignoring* the annotation is automatically immune to this drift. So index-only
  is the uniquely correct policy — exactly today's behavior, and the Rust kernel's.

** =wellFormed=-once suffices (the construction-site audit + 2 conditions)

Inference only ever sees (a) subterms of the validated body, or (b) terms built by
annotation-preserving =subst= / =whnf= (the [[#subst-lemma][lemma]]) or as =Forall= / =App=
wrappers over already-=WA= pieces. The audit of every fresh-=BVar= construction site:

- =App=-result =inst1(cod, a, 0)= (=Infer.lean:92,100=), =Let= zeta (=Infer.lean:119=),
  =Lam=-result =Forall(ty, infer(body))= (=Infer.lean:108=): all =WA=-preserving. ✓
- =expr_lift= / =expr_inst1='s own =BVar= constructions (=Subst.lean:158,214,272=): these
  *are* the lemma's subject. ✓
- *Condition 1 (required).* The η-expansion fresh =BVar(0)= at =DefEq.lean:497= must be
  annotated =BVar(0, lift(ty_a, 1, 0))=. It can be inferred directly (via =k_check= on the
  eta argument), so a missing/bogus annotation would be unsound. Mechanical, local fix.
- *Condition 2 (recommended hardening).* The pervasive =(0, store(BVar(0)))= None-sentinels
  in =Whnf.lean= / =Primitive.lean= (e.g. =Whnf.lean:204,344,855=) are *never read as live
  terms* — every consumer gates on the companion =G == 1= tag first (verified). Any dummy
  annotation is therefore dead data, but replacing the sentinel with a distinct
  obviously-dead constant prevents a future loosened =match= from ever inferring an
  ill-annotated =BVar=.

** The =wellFormed= check is structural, and the induction is well-founded
:PROPERTIES:
:CUSTOM_ID: wf-structural
:END:

Structural (pointer) equality against the recomputed =lift(Γ[i], i+1, 0)= is *sufficient
and correct*: inference will *use* the stored annotation as =infer(BVar(i,ty)) = ty=, so it
must be the *literal* canonical type, not merely a =def_eq= variant — keeping the tight
"equal index ⟹ pointer-equal annotation" invariant the completeness argument above relies
on. It is cheap (the same =lift= the kernel already runs, memoized; O(1) pointer compare),
needs no =whnf= / =def_eq=, and is well-founded: =wellFormed= validates a binder type
=τ= *before* pushing it and validating any =BVar= that points at it, and the annotation
=ty= is a structurally smaller subterm validated in the same pass.

** Verdict

*Sound, given Condition 1* (annotate the η =BVar(0)=) *and Condition 2 as hardening.* The
annotation is a memoized copy of a value the kernel already computes (=types_lookup=);
substitution transports it correctly at the right cutoff (proven, with a dependent trace);
the lifted form is necessary for local transport; index-only =def_eq= is sound and maximally
complete; one structural =wellFormed= pass — an extension of the existing
=validate_expr_well_scoped= — makes every downstream-seen term well-annotated. This is an
incremental tightening of an established, trusted pattern, not a new trust assumption.

* Memoization analysis
:PROPERTIES:
:CUSTOM_ID: memoization
:END:

#+begin_quote
The typed-BVar scheme *matches =ctx_trim='s sharing exactly* — no regression — while
deleting the context argument and the entire =ctx_trim= subsystem. This is strictly better
than the scrapped FVar scheme, which lost sharing on reused open subterms.
#+end_quote

** Sharing is preserved (the key advantage over FVar)

de Bruijn indices are *outer-invariant*: =BVar(0)= means "nearest binder" regardless of
absolute depth. So =BVar(0, ty)= is the *same node* at every depth where the nearest binder
has type =ty= — it shares across depths. The earlier FVar scheme used a depth-dependent
"compressed level" index, so the same opened leaf at different depths became *different*
nodes and *stopped* sharing. The typed-BVar scheme has no such regression.

Concretely, both schemes discriminate occurrences by exactly the same data — the relevant
binder types:

| Term shape                                       | =ctx_trim=                       | typed-BVar                          | Result |
|--------------------------------------------------+----------------------------------+-------------------------------------+--------|
| Closed subterm                                   | keys to =(e, Nil)=               | keys to =(e)= (annotations closed)  | wash   |
| Open subterm, same binder types                  | keys to =(e, [τ…]_ptr)= — shares | annotations identical → =e= shares  | wash   |
| Same open subterm at different /depths/          | shares (suffix identical)        | shares (index outer-invariant)      | wash   |

So sharing is a *wash with =ctx_trim=* across the board — but achieved with the context
*inside* the term, so the =types_ptr= field and the =ctx_trim= fixpoint disappear.

** The unambiguous wins

1. *Delete =ctx_trim= / =ctx_reachable= / =ctx_reachable_fix= / =ctx_reachable_scan=*
   (=Subst.lean:94-128=) and =types_lookup= (=Infer.lean:34-45=). The Rust mirror calls
   this family a *dominant* lookup overhead (=tc.rs:176-188=). In Aiur each of its
   sub-calls is a real memoized function (witness rows + lookup arity). Pure deletion.
2. *Drop the =types= column* from ~40 mutually-recursive functions — one fewer witness
   column and one fewer logUp-fingerprint field per call, kernel-wide.
3. *Self-contained expressions* — the enabling change that lets =whnf= / =def_eq= / the
   inductive checks also go context-free.

** The cost (be honest)

Annotations make terms *bigger*: every variable occurrence now carries a (lifted) type,
whose own variables carry types, nesting to the context depth. Content-addressing dedups
identical lifts, but distinct lifts (different depths) are distinct nodes, and =subst=/
=lift= now recurse into annotations. Versus =ctx_trim=, which stores each binder type
*once* in a shared context list, typed-BVars *replicate* the lifted types across
occurrences. For deeply dependent contexts this could be a meaningful blow-up in node
count / trace rows. *This is the one quantity to measure before committing*
([[#open-questions]]).

* Prior art
:PROPERTIES:
:CUSTOM_ID: prior-art
:END:

- *The validator already exists.* =validate_const_well_scoped= →
  =validate_expr_well_scoped= (=Check.lean:119-167=, Rust =check.rs:423-515=) already runs
  once per constant before inference, threading depth and rejecting loose BVars. =wellFormed=
  is this pass plus the annotation check. We are *extending* a trusted pass, not adding a new
  trust boundary.
- *The Rust kernel reaches the same goal by a different mechanism.* =src/ix/kernel/= opens
  binders into *free variables* (=expr.rs= =FVar(FVarId, Name, ExprInfo)=) whose types live
  in a =LocalContext= (=lctx.rs=), so its =infer= also needs no threaded context — but the
  type lives *beside* the term (in =lctx=), not *inside* it. The typed-BVar design keeps
  standard de Bruijn structure and puts the type *inside* the node. Same principle ("a
  variable knows its own type, so inference needs no context"); different mechanism.
- *Implication for testing*: the two kernels share *verdicts* but not memo structure, so
  differential testing compares accept/reject outcomes, not cache behavior.
- The =KValNode= / =KVal= / =KValEnv= NbE substrate (=KernelTypes.lean:75-96=) is *dead
  code* (=grep= finds it only in its own definition). Delete or mark vestigial; it is not
  prior art for this design.

* Alternatives considered
:PROPERTIES:
:CUSTOM_ID: alternatives
:END:

** FVar opening (the earlier, scrapped proposal)
:PROPERTIES:
:CUSTOM_ID: alt-fvar
:END:

The previous draft proposed *opening* binders into fresh free variables =FVar(idx, type)=,
with the fresh index computed context-free as =fvar_ceiling(e) = 1 + max FVar index=, plus a
new =expr_close= primitive, and — because two independently-opened terms could collide — a
*joint* opening rule in =def_eq= (=k = max(ceiling(a), ceiling(b))=). It is sound, but only
under five delicate conditions (joint =max= at *every* binder arm incl. η; index-only FVar
equality; an =expr_close= that also lifts loose bvars; …). Typed BVars are *strictly better*:

- *No special comparison logic.* =def_eq= stays plain structural index equality; the entire
  joint-opening apparatus (and its five conditions) vanishes.
- *No new primitives / no open-close dance.* Standard de Bruijn is preserved; only the node
  gains a field, and =subst= recurses into it.
- *No memoization regression.* FVar's compressed-level index is depth-dependent and loses
  sharing on reused open subterms; de Bruijn indices are outer-invariant and don't.
- *Wire format unchanged* (FVar needed a serialization refusal; it's an internal-only node).

The cost typed BVars pay that FVar did not: annotations are carried on *every* variable
occurrence (FVar carried the type once per opened binder). That is the trade — paid in term
size, bought back in simplicity and sharing.

* Migration plan
:PROPERTIES:
:CUSTOM_ID: migration
:END:

1. *Stage 0 — node + subst + matches (mechanical).* Change =BVar= to =BVar(G, KExpr)=.
   Update the ~18 match arms (bind the field). Make =subst= (=expr_lift=/=expr_inst1=/
   =expr_inst_many=/=expr_lbr=/=expr_inst_levels=) recurse into the annotation
   ([[#subst-rules]]). Give the ~150 construction sites a *placeholder* annotation. Compiles;
   semantics not yet relied upon.
2. *Stage 1 — ingress.* Thread the binder-type context through =convert_expr= and emit real
   annotations (=Convert.lean=). Annotate the η site (=DefEq.lean:497=, Condition 1) and
   replace the None-sentinels with a dead constant (Condition 2).
3. *Stage 2 — =wellFormed=.* Extend =validate_expr_well_scoped= to thread binder types and
   check =ty ≡ptr lift(Γ[i], i+1, 0)=. It already runs at =check_const=.
4. *Stage 3 — atomic cluster cutover.* Switch =k_infer= (=BVar(i,ty) => ty=; no context
   push), =whnf=, =k_is_def_eq=, the inductive checks and ~40 helpers to drop =types=. The
   cluster is mutually recursive — one change.
5. *Stage 4 — delete dead machinery.* Remove =ctx_trim= / =ctx_reachable*= / =types_lookup=.
6. *Rust*: stays FVar-based; align as a *verdict* oracle for differential testing.

* Testing strategy
:PROPERTIES:
:CUSTOM_ID: testing
:END:

- *Differential oracle*: run the existing corpus + heavy mathlib blocks through old
  (=types=-threaded) and new (typed-BVar) kernels; assert identical =infer= / =whnf= /
  =def_eq= verdicts. Use the Rust kernel as a second verdict oracle.
- *Substitution-preservation property test*: for random well-annotated terms and
  substitutions, assert =wellFormed(subst(t, …))= holds — directly exercising the
  [[#subst-lemma][lemma]], especially dependent binder types whose own bound variables get
  substituted (where an off-by-one would hide).
- *Adversarial =wellFormed=*: feed terms with deliberately *wrong* annotations and loose
  BVars; assert rejection.
- *η + proof-irrelevance*: the η site's annotated =BVar(0)= (Condition 1) inferred via
  =k_check=; =try_proof_irrel= / unit-like / struct-eta Prop-guards re-entering context-free
  =infer= on open annotated subterms.
- *Sentinel safety*: assert no =(0, BVar(0))= sentinel ever reaches =infer= / =def_eq= live
  arms (Condition 2).
- *Memoization metrics (honest)*: confirm sharing matches the =ctx_trim= baseline, and
  *measure the annotation-size blow-up* (node count / trace rows) on deeply dependent terms.

* Open questions
:PROPERTIES:
:CUSTOM_ID: open-questions
:END:

1. *Annotation size — how bad?* Typed BVars replicate lifted types across occurrences and
   nest them to context depth. *Measured* (gb/typed-bvar, full ixvm suite): a modest FFT
   increase — HEq −0.1%, Nat ~0%, Nat.add +0.4%, Nat.add_comm +3.5%, Array.append_assoc
   +2.6%, Even.rec +0.8%, and the variable-/recursion-dense terms most: Nat.decLe +10%,
   nat_div_lit +11%, str_size_lit +7%. So the annotation cost dominates the (small)
   sharing gain over the old =ctx_trim= (which already shared closed terms): the refactor
   is an *architectural* win (context-free kernel, deleted =ctx_trim=, expr-pointer-only
   memo key), not a raw FFT reduction. Mitigations if the increase matters: intern
   annotations aggressively; or store a *compressed* annotation (only when the type isn't
   cheaply recomputable) — at the risk of reintroducing context. The current design stores
   the full lifted type verbatim.
2. *Should =wellFormed= also =k_ensure_sort= binder types*, or is the existing in-inference
   =k_ensure_sort= sufficient? (Leaning: keep them separate — =wellFormed= checks annotation
   /canonicality/, inference checks binder /validity/; they compose.)
3. *Sentinel representation.* Adopt a single distinguished dead =KExpr= for all
   None-markers (Condition 2), or annotate each in place? A shared dead constant is cleaner
   and self-documenting.
4. *Ingress vs. wellFormed division of labor.* If ingress always emits canonical annotations,
   =wellFormed='s annotation check is a pure *re-derivation-and-compare* (defends against
   adversarial/non-deterministic witnesses). Confirm the threat model: are annotations ever
   trusted from an untrusted source without =wellFormed= running first? (They must not be.)
5. *Rust convergence.* Worth moving Rust from FVar+lctx to typed BVars for memo-structure
   parity, or keep verdict-only differential testing? Likely the latter — Rust isn't
   memo-constrained the way Aiur is.
